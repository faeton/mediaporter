## 2026-05-14 ‚Äî `SyncAllowed` is NOT terminal-equivalent to `SyncFinished`

Symptom: large-file syncs (Violet HEVC: 1.5 GB transcode, 6 GB original) landed the row in TV.app but the row was unbound (`base_location_id=0`, `location=''`, `file_size=0`) and the bytes vanished from `/iTunes_Control/Music/Fxx/...mp4` within seconds. Mini-clip (141 MB, 4 s upload) worked perfectly with identical metadata + flags.

Initial wrong hypothesis: `Variant = { AssetParts = 3 }` in AssetManifest implied medialibraryd expected V + alt-audio + sub sidecars and got just one combined MP4. Disproven ‚Äî mini-clip got `AssetParts = 3` too and bound fine. The flag is informational, not a contract.

Real cause (found via side-by-side log diff): `ATCSession.finishSync` treated `SyncAllowed` as terminal-equivalent to `SyncFinished`. The device sends `SyncAllowed` early in the post-MetadataSyncFinished phase (right after FileBegin / AssetManifest exchange) as "you may proceed". During a long upload it accumulates in the drainer inbox alongside `InstalledAssets` / `AssetMetrics`. When finishSync starts after FileComplete, it grabs the stale `SyncAllowed` immediately (via `drainInbox()` in insertion order) and returns. medialibraryd never gets the chance to commit our asset ‚Üí row stays unbound ‚Üí background sweep deletes the orphan file.

Why mini-clip always worked: upload finished in <4 s, so `SyncFinished` and the early `SyncAllowed` both landed within ~35 ms of FileComplete. `SyncFinished` was first in the inbox, `finishSync` accepted it.

Side-by-side log evidence (single MacApp session, same anchor sequence):

| File | Upload | finishSync result | TV.app outcome |
|------|--------|-------------------|---------------|
| mini-clip 141 MB | 4 s | `via=SyncFinished elapsed=0s` | plays |
| Violet 1.5 GB | 49 s | `via=SyncAllowed elapsed=0s` (stale `SyncAllowed`+`InstalledAssets`+`AssetMetrics` in inbox) | row visible, `base_location_id=0`, file swept |
| Violet 6 GB (post-fix) | 3:06 | `via=SyncFinished elapsed=0s` (no early `SyncAllowed` arrived this run) | plays |

Fix: `finishSync` (1) drops anything already in inbox on entry (stale upload-phase noise), (2) waits strictly for `SyncFinished` for up to 120 s, (3) falls back to `SyncAllowed` only after a 30 s grace period with no `SyncFinished` (safety net for misbehaving devices). New log markers: `atc.finishSync.discard_stale`, `atc.finishSync.syncallowed`, `atc.finishSync.done via=SyncFinished|SyncAllowed_fallback`.

Likely also explains the open "re-uploaded deleted episode doesn't bind on first sync" item from commit 00450b4 ‚Äî same family of symptoms, same root cause class.

## CLI Improvements (2026-04-10) ‚Äî Python v0.3.1

Real-world two-file run (Send.Help 4.7 GB + Avatar 15 GB) exposed and fixed:

1. **ffmpeg stderr pipe deadlock** ‚Äî long transcodes froze partway. Cause: `subprocess.Popen(stderr=PIPE)` with only stdout drained; once ffmpeg's stderr filled the ~64 KB OS pipe, ffmpeg blocked on `write(2)`. Fix: background stderr-drain thread in `transcode.py` with rolling 200-line tail.
2. **Ctrl+C didn't kill ffmpeg** ‚Äî `ThreadPoolExecutor.__exit__` blocked on workers stuck on ffmpeg output. Fix: module-level Popen registry + `cancel_all()`; `transcode_and_sync` / `transcode_all` catch `KeyboardInterrupt` ‚Üí cancel pending futures ‚Üí `cancel_all()` ‚Üí executor shutdown.
3. **Sequential upload waited for all transcodes** ‚Äî old `transcode_all ‚Üí sync_all` not pipelined. Fix: `transcode_and_sync()` runs a single-threaded uploader alongside the ThreadPoolExecutor. Split `sync_files()` into `make_sync_file_info()` + `afc_upload_one()` + `register_uploaded_files()`.
4. **Mixed-codec files always re-encoded to AAC** ‚Äî wasteful (AC3/EAC3 play natively). Fix: `pick_normalization_codec()` picks best codec already in file (EAC3 > AC3 > AAC), uses `-c:a copy` for matching tracks. `_partition_jobs` promotes mixed-codec files to `needs_remux=True` even for MP4.
5. **No disk-space preflight** ‚Äî could fill `/tmp` or device mid-run. Fix: `_check_disk_space()` queries `shutil.disk_usage(tempfile.gettempdir())` for Mac and `query_device_disk_space()` for device; compares against `sum(source_sizes) * 1.1`.
6. **No run summary** ‚Äî added `PipelineStats` + `_print_summary()` (wall clocks, bytes, peak/avg speed, free-space deltas).
7. **`mediaporter` with no args printed --help** ‚Äî now routes `[]` ‚Üí `["sync"]` ‚Üí `prompt_for_files()`.
8. **`devices` showed only UDID** ‚Äî now queries lockdown `AMDeviceCopyValue` for DeviceName/ProductType/etc. and prints friendly model + native resolution + recommended target. ~60 ProductType mappings in `sync/device.py`.

## Resolved questions

- **How to set media_type=2048?** ‚Äî `is_movie: True` + `location.kind: "MPEG-4 video file"` in insert_track plist. Airlock staging NOT required.
- **Is Airlock staging needed?** ‚Äî NO. Single upload to `/iTunes_Control/Music/Fxx/` is sufficient.
- **Are media_type/media_kind settable via sync plists?** ‚Äî YES, with `is_movie` + `location.kind`.
- **Why do large file syncs fail?** ‚Äî Two causes: (1) ATC session timeout if MetadataSyncFinished sent after long upload ‚Äî send before. (2) Ping/Pong keepalive ‚Äî must respond or session drops.
- **Why does device never send SyncFinished?** ‚Äî Stale pending assets. Device waits for ALL assets in AssetManifest. Send FileError for stale ones.
